# ПолБака — план проекта и дорожная карта

> Краудсорсинговый сервис наличия топлива на АЗС с упором на **достоверность данных**.
> Название — **ПолБака** (проверено 02.07.2026: ниша свободна; домен polbaka.ru — проверить при регистрации).
> Прежнее имя BenzRadar отброшено: обнаружен конкурент benzinradar.com («БензинРадар»).
> **Старт — Санкт-Петербург + Ленинградская область.**

---

## 1. Проблема и цель

В 2026 в РФ перебои с топливом на АЗС. Водителю нужно знать **прямо сейчас**:
есть ли на конкретной заправке нужная марка, какая очередь, есть ли лимит на отпуск.

Уже есть конкурент — **gdebenz.ru**: карта, отметки юзеров, фильтры по топливу, статусы «есть/очередь/пусто», маршрут. Сделано добротно.

**Наша цель — не копия, а продукт, которому можно доверять и который сам тебя оповещает.**

## 2. Чем берём (отличия от gdebenz.ru)

| Их слабое место | Наше решение |
|---|---|
| Анонимные отметки быстро протухают, легко наврать | **Умное ядро доверия**: свежесть с затуханием, вес по репутации, двойное подтверждение |
| Надо самому зайти и проверить | **Telegram-бот**: подписка на заправку → пуш «появился 95-й» |
| История отметок не используется | **Прогноз завоза** (фаза 2): «обычно завозят по утрам» |
| Никто не отслеживает качество топлива, только наличие | **Сигнал о качестве** (отдельно от доступности): жалобы после заправки, агрегация с порогом, без вердикта — см. п.6.5 |

Приоритет — **умная часть** (доверие + прогнозы). Красивая карта вторична, на старте минимально функциональная.
Цель — **реальный запуск в прод**, начиная с одного города.

## 3. Архитектура

```
[ Веб-карта (Leaflet) ]        [ Telegram-бот (aiogram) ]
            \                          /
             \                        /
              v                      v
            [  FastAPI  — REST API + умное ядро  ]
                          |
                          v
        [ PostgreSQL + PostGIS ]  <-- гео-запросы, отметки, история
                          ^
                          |
        [ Импортёр АЗС из OpenStreetMap (Overpass API) ]  (разовый/периодический)
```

## 4. Технологический стек

| Слой | Технология | Почему |
|---|---|---|
| API-бэкенд | **FastAPI** (Python, async) | Async-стиль как в aiogram; родной для asyncpg |
| База | **PostgreSQL + PostGIS** | Уже знаком с Postgres; PostGIS — гео («ближайшие в радиусе») |
| Доступ к БД | **asyncpg** (+ возможно SQLAlchemy позже) | Уже используется в текущем боте |
| Карта / фронт | **Leaflet + OpenStreetMap** | Бесплатно, без API-ключей и лимитов |
| Бот | **aiogram 3.x** (новый, отдельный) | Переиспользуем навык; «Хлеб не забудь» не трогаем |
| Источник АЗС | **Overpass API** (OSM, `amenity=fuel`) | Бесплатный список заправок РФ с координатами |
| Деплой | VPS + Docker (уточнить в фазе 3) | Прод-запуск |

## 5. Схема базы данных (черновик)

**stations** — заправки (из OSM)
- `id` (PK), `osm_id`, `name`, `brand`, `lat`, `lon`, `geom` (PostGIS point)
- `address`, `city`, `region`, `created_at`

**users** — пользователи (веб и бот)
- `id` (PK), `tg_id` (nullable), `reputation` (float; аноним ≈ 0.3, залогиненный старт 1.0), `is_anonymous` (bool), `created_at`

**reports** — отметки о наличии
- `id` (PK), `station_id` (FK), `user_id` (FK)
- `fuel_type` (enum: AI92, AI95, AI98, DT)
- `status` (enum: available, low, queue, empty)
- `queue_min` (nullable), `limit_liters` (nullable)
- `comment` (nullable — необязательный свободный текст поверх шаблона)
- `created_at`, `source` (web/bot)

> **Модель отметок — из шаблона:** марка/статус/очередь вводятся кнопками (строгие значения),
> чтобы умное ядро могло считать доверие и строить прогнозы. Свободный текст — только опциональный `comment`.

**station_status** — агрегированный текущий статус (пересчитывается)
- `station_id` + `fuel_type` (PK), `confidence` (float), `status`, `updated_at`

**subscriptions** — подписки бота
- `id` (PK), `user_id` (FK), `station_id` (FK), `fuel_type`, `created_at`

**quality_reports** — отчёты о качестве топлива (отдельная сущность, не поле в `reports`)
- `id` (PK), `station_id` (FK), `user_id` (FK, **обязателен** — анонимные жалобы о качестве не принимаются)
- `fuel_type` (тот же enum, что у `reports`)
- `fueled_at` — когда пользователь, по своей памяти, заправился (отдельно от `created_at` — момента жалобы; между ними может быть разрыв в дни)
- `symptom` (enum: детонация/стук, троит/глохнет, потеря мощности, другое) — строгие значения кнопками, как и у `reports`
- `comment` (необязательный свободный текст поверх шаблона)
- `created_at`

> **Не путать с `reports`:** `reports` — «что сейчас на станции» (TTL часы, транзитный факт). `quality_reports` — утверждение о прошлом событии, всплывающее спустя дни, с принципиально иной ценой ошибки (репутационный урон конкретному бизнесу, а не самоисправляющийся факт). Подробности — п.6.5.

## 6. Умное ядро — сердце проекта

Каждая отметка не абсолютна, а имеет **вес доверия**:

```
confidence(station, fuel) = Σ [ freshness(t) × reputation(user) ] по отметкам
```

- **freshness(t)** — экспоненциальное затухание со временем.
  Напр. period = 60 мин: отметка 5 мин назад ≈ 0.94, 2 часа назад ≈ 0.13.
- **reputation(user)** — вес автора. Аноним (без входа) — фиксировано низкий вес ≈ **0.3**; залогиненный через Telegram — старт 1.0, растёт при подтверждениях, падает за противоречия.
- **Гибридная модель входа:** отмечать можно **без авторизации** (низкий вес), но на карточке видно «обновил аноним» — доверия визуально меньше. Вход через Telegram даёт больше веса и репутацию.
- **Двойное подтверждение** — статус `available` показываем «уверенно» (зелёный), только если суммарный confidence выше порога (≈ 2 свежих независимых отметки).

**Отображение свежести на карте:**
- < 15 мин и подтверждено → яркий цвет статуса
- 15–60 мин → приглушённый
- \> 60 мин → серый «данные устарели, проверьте»

**Прогноз (фаза 4)** — из истории `reports` ищем паттерны времени завоза по каждой АЗС.

## 6.5 Качество топлива — отдельный сигнал (не путать с доступностью)

Помимо «есть/нет топлива» собираем сигнал о **качестве**: заправился → через день-два заметил симптомы (троит, детонация) → сообщил через бота.

**Ключевые сложности (не «просто добавить поле»):**

- **Временная атрибуция.** Три момента времени — заправка, симптом, жалоба — разнесены на дни; пользователь мог заправиться ещё где-то в промежутке. Это не решается схемой: данные принципиально *сигнал*, а не *факт*. `fueled_at` хранится отдельно от `created_at`, чтобы можно было понижать вес жалоб с большим лагом.
- **Субъективность симптомов.** Проблемы двигателя имеют десятки причин, не связанных с топливом (свечи, форсунки, погода, стиль езды). Единичная жалоба — слабое свидетельство → нельзя показывать что-либо по одной жалобе.
- **Накрутка / клевета конкурентами.** У жалобы на качество несравнимо выше мотивация для атаки на конкурента и дольше живущие последствия, чем у отметки об очереди. Это главный риск фичи, не побочный.
- **Юридический риск.** Публично связать конкретную АЗС с «продают паленый бензин» — это утверждение о вине, а не нейтральный факт вроде «нет топлива». Может квалифицироваться как клевета/недобросовестная конкуренция, если жалобы окажутся необоснованными или скоординированной атакой. **Перед публичным включением именно этой фичи — сверка с юристом.** Разработку и внутреннее тестирование это не блокирует, публичный запуск — блокирует.

**Агрегация — переиспользуем паттерн confidence (п.6), с поправками:**

- Без материализованной `station_status`-таблицы для качества — считаем на лету по скользящему окну (например, 30–60 дней).
- Порог показа — **несколько независимых пользователей** (не жалоб — иначе один аккаунт «накачивает» станцию) с похожим симптомом в ограниченном окне. Константы заметно консервативнее, чем у доступности: ориентир — **не меньше 3 независимых юзеров**.
- На карте — **не вердикт**, а формулировка вида *«N пользователей сообщили о возможных проблемах за последние M дней»*. Никаких оценок вида «качество: 2/10» — разница между «агрегат отчётов» и «утверждение факта» важна и продуктово, и юридически.

**Защита от злоупотреблений:**

- Обязательная авторизация через Telegram — анонимные жалобы о качестве не принимаются (в отличие от `reports`, где аноним разрешён с низким весом).
- Rate-limit на пользователя/станцию.
- Полный аудиторский след (кто/когда/что) хранится всегда, даже если публично не показывается — на случай спора со станцией.
- Ручная модерация на старте **не обязательна** — порог «≥3 независимых пользователя + окно времени + только залогиненные» достаточен как автоматический MVP-фильтр. Возвращаемся к модерации только если увидим реальные злоупотребления.

## 7. REST API (черновик эндпоинтов)

- `GET  /stations?bbox=...&fuel=AI95` — заправки в области карты со статусами
- `GET  /stations/{id}` — карточка АЗС + последние отметки
- `POST /reports` — отправить отметку `{station_id, fuel_type, status, queue_min?}`
- `GET  /stations/{id}/history` — история для графика/прогноза
- `POST /quality-reports` — жалоба о качестве `{station_id, fuel_type, fueled_at, symptom, comment?}` (только для авторизованных)
- `GET  /stations/{id}/quality` — агрегированный сигнал качества за окно, формат «N сообщили за M дней», без вердикта
- (бот) `POST /subscriptions`, `DELETE /subscriptions/{id}`

## 8. Бот (aiogram, новый)

- `/start` → выбор города/геолокация
- «Найти рядом» → ближайшие АЗС со статусами
- «Следить за АЗС» → создаёт subscription
- Фоновая задача: при новой отметке `available` по подписке → шлёт уведомление
- «Отметить» прямо из бота → `POST /reports` (бот как ещё один источник данных)
- «Сообщить о качестве» → заправился здесь недавно, подозреваю плохое топливо → `POST /quality-reports` (только для авторизованных через бота)

## 10. Стратегия запуска (борьба с холодным стартом)

Краудсорс не работает без критической массы отметок. План:
- Стартуем в **одном городе** (выбрать свой/крупный проблемный).
- **Посеять данные вручную** на старте, позвать знакомых-водителей.
- Бот даёт ценность даже при малом объёме (подписался — получил пуш).
- Локальные чаты/сообщества автомобилистов для первого притока.
